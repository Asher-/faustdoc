Using and Developing Architectures Files

Introduction

A Faust program describes a signal processor, a pure DSP computation that maps input signals to output signals. It says nothing about audio drivers or controllers (like GUI, OSC, MIDI, sensors) that are going to control the DSP. This additional information is provided by architecture files. We assume here a general understanding of the architecture files model. In this tutorial, we'll explain how the machinery works:

  • starting with a simple example

  • compter des buttons et sliders: faust -a architecture (renseigner sur le DSP), ou générer un fichier de sortie. example avec JSONUI

  • gradually enriching it to show how multiple controllers can be used (OSC, http, MIDI etc.)

  • showing how the architectures files can be used with statically generated C++ files, or dynamically generated DSP using the LLVM or Interpreter backends

  • demonstrate how to create polyphonic instruments, add a global effects using the DSP C++ algebra, setup sample accurate controllable DSP

  • finally demonstrating how to write simple architecture files, or combine, extend and use existing ones.

  • expliquer: faust2xx

  • generation de code (exemple swift sans passer par le JSON)

#define FAUSTFLOAT float

class dsp {};

struct Meta {
    virtual void declare(const char* key, const char* value) {};

struct Soundfile {
    FAUSTFLOAT** fBuffers;
    int* fLength;   // length of each part
    int* fSR;       // sample rate of each part
    int* fOffset;   // offset of each part in the global buffer
    int fChannels;  // max number of channels of all concatenated files
};

struct UI {

    virtual void openTabBox(const char* label) {}
    virtual void openHorizontalBox(const char* label) {}
    virtual void openVerticalBox(const char* label) {}
    virtual void closeBox() {}

    virtual void addCheckButton(const char* label, FAUSTFLOAT* zone) {}
    virtual void addVerticalSlider(const char* label,

    virtual void addHorizontalSlider(const char* label,

    virtual void addNumEntry(const char* label,

};

<<includeclass>>

When used with the -a option, the compiler will simply replace them with the C++ class (or C code when used with the C backend, or any other textual output when used with backend like Rust, SOUL...).

Type the following command to generate a ready to compile C++ file:

faust -a nullarch.cpp test1.dsp -o test1.cpp

And compile it with, using the -S option since the file does not even contain a main and would not compile otherwise:

c++ -S test1.cpp

Using the GTKUI controller

The GTKUI class uses the GTK toolkit to create a Graphical User Interface with a proper group-based layout. It can simply be plugged on the DSP using buildUserInterface a second time on the DSP:

#include "faust/gui/GTKUI.h"

// Create and plug the GTK interface
GTKUI gtk_interface(name, &argc, &argv);
DSP->buildUserInterface(&gtk_interface);

// Start gtk_interface (layout and event handing)
gtk_interface->run();
  • use a ready-to-use GUI architecture: GTKUI using GTK framework
  • explain and demonstrate MapUI (with setParamValue/getParamValue model), explain the label and path distinction, should the PathBuilder helper.

Using the OSCUI controller

The OSCUI controller can be used to change parameters with incoming OSC messages, or emit OSC messages when parameters are moved. It can simply be plugged on the DSP using buildUserInterface a third time with the following code:

#include "faust/gui/OSCUI.h"

// Create and plug the OSC interface
OSCUI osc_interface(name, &argc, &argv);
DSP->buildUserInterface(&osc_interface);

// Start osc_interface
osc_interface->run();

Now received OSC messages on the correct path and input port will change the parameters values, and moving the GUI items (button, sliders...) will generate OSC messages on the output port. This synchronization mechanism between the two controllers is implemented in a generic way in the GUI class, which both GTKUI and OSCUI are sub-classes of.

Using the httpdUI controller

  • use httpdUI, as a second example of remote control

The httpdUI class implements HTTP remote control using the libmicrohttpd library to embed a HTTP server inside the application. Then by opening a browser on a specific URL (http://127.0.0.1:5510), the GUI will appear and allow to control the distant application or plugin. The connection works in both directions.

#include "faust/gui/httpdUI.h"

// Create and plug the Httpd interface
httpdUI httpd_interface(name, &argc, &argv);
DSP->buildUserInterface(&httpd_interface);

// Start httpd_interface
httpd_interface->run();

Since thehttpdUIclass is also a sub-class of the GUI class, all UI items stay synchronized.

Using metadata

The Faust language allows widget labels to contain metadata enclosed in square brackets as key/value pairs. These metadata are handled at GUI level by a declare method taking as argument, a pointer to the widget associated zone, the metadata key and value:

declare(FAUSTFLOAT* zone, const char* key, const char* value);

Lets take a first concrete example with unit The [midi:ctrl 7] metadata can be added.

Lets take a second concrete example with MIDI metadata to control sliders. The [midi:ctrl 7] metadata can be added on a slider to control its value with a MIDI CtrlChange 7 event.

The MIDI metadata are decoded using the MIDIUI class, initialized with a midi_hander class to be connected to real MIDI devices. In the following piece of code, a MidiUI object is created and connected to a rt_midi MIDI messages handler (using the RTMidi library), then given as a parameter to the standard buildUserInterface to control DSP parameters:

...
rt_midi midi_handler("MIDI");
MidiUI midi_interface(&midi_handler);
DSP->buildUserInterface(&midi_interface);
